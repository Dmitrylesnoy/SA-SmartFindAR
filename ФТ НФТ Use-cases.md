## Функциональные требования

1. FR

  **ID: CR-1**

  **Название:** Создание и сохранение предмета в хранилище

  **Акторы:** 

    - Пользователь

    - Приложение 

    - Сервер 

  **Предусловия:** 

    - У пользователя есть интернет подключение

    - У пользователя работает камера на смартфоне

  **Основной поток:** 

    1. Пользователь открывает меню создания карточки предмета

      В меню создания карточки предмета есть поля для заполнения:

      1. Фотографии 

      2. Фото на месте

      3. Название

      4. Описание (необязательно)

    2. Пользователь нажимает кнопку камеры для создания фото для поля "Фотографии"

    3. App показывает пользователю предупреждение, что в кадре должно быть видно предмет. App открывает камеру смартфона

    4. Пользователь фотографирует предмет — фотография прикрепляется к полю фотографии в меню создания карточки предмета

    5. Пользователь повторяет шаг 2, 3, 4 для поля "Фото на месте". App показывает предупреждение "В кадре должен быть виден предмет и его окружение"

    6. Пользователь заполняет поле Название

    7. Пользователь нажимает кнопку "Сохранить"

    8. App отправляет на сервер данные о предмете,

    9. Сервер обрабатывает фото предмета, определяет местоположение. 

    10. Сервер сохраняет данные о предмете в хранилище пользователя

    11. App убирает меню создания предмета и показывает список предметов пользователя 

  **Постусловия:** 

    Успех:

      1. App показывает в списке предметов созданную карточку предмета 

      2. В хранилище пользователя сохранены данные о созданном предмете

        1. Данные заполненные в карточке пользователем: фотографии, фото места, название, описание

        2. Определённое местоположение предмета

    Неудача:

      1. Пользователю показывается предупреждение об ошибке

      2. Пользователь возвращается в меню создания карточки предмета

      3. Данные о предмете не сохраняются

  **Альтернативные потоки:** 

    Сервер не смог определить местоположение предмета

    Точка входа: Шаг 9

    9.1. Сервер не смог определить местоположение предмета по полученным фото

    9.2. Сервер отсылает ответ о негативном результате обработки

    9.3. Пользователь получает сообщение о результате обработки фото, данные не сохраняются

    9.4. App возвращается в меню создания карточки предмета. Все поля кроме фото сохранены предыдущим вводом, поле Фото — сброшено (не заполнено)

  **Исключительные потоки:** 

    Потеря интернет соединения при отправке данных на сохранение

    Точка входа: Шаг 8

    8.1. При отправке данных, сигнал об успешном получении данных с сервера не пришел/пришел сигнал о неуспешном получении

    8.2. Пользователю показывается сообщение: "Ошибка соединения"

    8.3. App остаётся в меню создания карточки предмета с заполненными значениями полей

  **Бизнес правила:** 

    1. Сохранённый предмет доступен его создателю и тем, с кем пользователь поделился доступом

    2. Сохранить предмет в хранилище пользователя может только он сам создав его, или получив доступ

2. Func req 

  **ID: SC-1**

  **Название:** 3D ****сканирование помещения

  **Акторы:** 

    1. Пользователь

    2. App приложение

    3. AR ядро смартфона

    4. Сервер

  **Предусловия:** 

    1. На телефоне есть AR ядро

    2. 3д скан помещения еще не производился

    3. Камера смартфона пользователя работает исправно

    4. У пользователя есть интернет подключение

  **Основной поток:** 

    1. Пользователь заходит в функцию сканирования помещения. Нажимает кнопку "Начать сканирование"

    2. App запускает AR ядро смартфона. Ядро инициализируется

    3. Ядро определяет поверхности и сканирует помещение

    4. App получает сырую 3д модель скана помещения  и проверяет её

    5. App отправляет модель на сервер, Сервер обрабатывает 3д модель. 

    6. Сервер сохраняет её в хранилище пользователя

    7. Сервер отвечает App успешным сохранением 3д скана

    8. App сохраняет 3д модель в локальной памяти

    9. Сообщает пользователю: "Успешное сохранение". Возвращается на страницу списка предметов

  **Постусловия:** 

    Успех:

      1. На сервере, в хранилище пользователя сохранена обработанная 3д модель помещения

      2. В App сохранена 3д модель помещения

    Неудача:

      1. Данные 3д модели сброшены. 

      2. App запускает функцию сканирования заново

  **Альтернативные потоки:** 

    Накопленная ошибка AR сканирования / дефект сканирования

    Точка входа: шаг 4

    4.1. App находит дефекты модели и сообщает негативный результат обработки пользователю

    4.2. Пользователь видит сообщение от App: "Неудачное 3д сканирование, попробуйте еще раз"

    4.3. Пользователь остаётся в меню сканирования, сканирование начинается заново

  **Исключительные потоки:** 

    Сбой AR ядра

    Точка входа: шаг 2, 3

    2.1. При инициализации и работе AR Ядра произошел сбой

    2.2. App останавливает AR ядро. Пользователь видит сообщение: "Ошибка сканирования, повторная попытка...". 

    2.3. Шаг 2 повторяется еще 2 раза в случае повторного появления ошибки.

    2.4. В случае неудачи пользователь видит сообщение "AR модуль неисправен". App возвращается в меню списка предметов

3. Func req 

  **ID: ED-1**

  **Название:** Изменение фотографии места предмета

  **Акторы:** 

    1. App

    2. Пользователь

    3. Сервер

  **Предусловия:** 

    1. У пользователя существует исправный скан помещения

    2. У пользователя сохранён предмет с заполненными полями и фотографиями

  **Основной поток:** 

    1. Пользователь открывает карточку предмета из списка.

    2. Пользователь нажимает "Изменить". App открывает меню редактирования предмета

    3. Пользователь нажимает кнопку фотографирования предмета на новом месте.

    4. App открывает камеру. Пользователь видит сообщение "В кадре должен быть виден предмет и его окружение"

    5. Пользователь делает снимок. App прикрепляет полученное фото в поле "Фото на месте"

    6. Пользователь нажимает "Сохранить". App отправляет изменённые данные на сервер.

    7. Сервер валидирует данные, определяет фото места предмета. Сохраняет данные в хранилище пользователя

    8. Сервер отправляет App результат обработки данных

    9. App показывает пользователю сообщение о результатах обработки данных сервером. Переходит в меню списка вещей

  **Постусловия:** 

    Успех:

      1. Новые данные о вещи, введённые пользователем сохранены на сервере в хранилище пользователя

      2. Сервер определил местоположение предмета по новому фото места и сохранил данные в хранилище пользователя

      3. В App открыто окно просмотра списка вещей

    Неудача:

      1. Данные предмета на сервере в хранилище пользователя не обновлены

      2. App остался на меню изменения предмета, обновлённые данные сброшены. Пользователь видит сообщение об ошибке

  **Альтернативные потоки:** 

    Сервер не смог распознать предмет по фотографии места

    Точка входа: Шаг 7

    7.1. Сервер не смог определить распознать предмет на полученным фото места

    7.2. Сервер отсылает ответ о негативном результате обработки, данные не сохраняются

    7.3. Пользователь получает сообщение о том что предмет не распознан

    7.4. App остаётся в меню редактирования карточки предмета. Ввод всех полей кроме фото места сохранился, Фотография места — прежняя (не обновлённая)

  **Исключительные потоки:** 

    Сервер недоступен

    Точка входа: шаг 6

    6.1. App не получает ответа от сервера

    6.2. App выводит пользователю сообщение: "Сервер временно недоступен, попробуйте позже"

    6.3. App переходит на окно просмотра списка предметов

  **Бизнес правила:** 

    1. Изменять данные предмета может владелец предмета\создатель

    2. Изменять данные предмета может пользователь, который получил доступ к предмету

4. Func req 

  **ID: SH-1**

  **Название:** изменение предмета в общем доступе

  **Акторы:** 

    1. Пользователь Владелец предмета

    2. Пользователь, с которым Владелец поделился доступом — Друг

    3. App

    4. Сервер

  **Предусловия:** 

    1. У владельца есть сохранённая вещь, доступом с которой он поделился с Другом

    2. Друг имеет доступ к вещи общего доступа (просмотр, изменение, сохранение)

    3. У Друга и Владельца есть интернет подключение

  **Основной поток:** 

    1. При открытии другом окна списка вещей, в нём отображается вещь, к которой получен доступ. Данные о вещи актуальны и синхронизированы с владельцем

    2. Друг открывает меню редактирования вещи общего доступа

    3. App отправляет на сервер запрос с проверкой доступности вещи в хранилище на изменение

    4. Друг изменяет данные вещи и нажимает "Сохранить"

    5. App отправляет серверу новые данные вещи

    6. Сервер валидирует, определяет место предмета (см CR-1) и обновляет данные о вещи в хранилище владельца

    7. App открывает меню просмотра списка вещей. Пользователь видит сообщение об успешном изменении.

  **Постусловия:** 

    Успех:

      1. Данные о вещи обновлены в хранилище владельца

      2. Новые данные доступны всем пользователям, имеющим доступ

    Неудача:

      1. Новые данные, введённые другом не сохранились на сервере

      2. Пользователю показано сообщение о произошедшей ошибке

      3. Пользователям, имеющим доступ видная прежняя версия данных вещи

  **Альтернативные потоки:** 

  1. Вещь общего доступа больше недоступна для друга

    Точка входа: Шаг 1.

    1.1 В списке вещей не отображается вещь общего доступа

  2. Вещь общего доступа стала недоступна для друга во время изменения

    Точка входа: Шаг 4

    4.1. Сервер отправляет App сообщение, что вещь больше недоступна. Сервер удаляет вещь из хранилища Друга

    4.2. App показывает пользователю сообщение, что вещь больше недоступна. App переходит в окно списка вещей. Вещь не отображается в списке

  с. Во время редактирования данные вещи были обновлены другим пользователем

    Точка входа: Шаг 4

    4.1 Сервер отправляет App сообщение, что данные вещи у App неактуальны, на сервере имеется более новая версия данных. Данные, полученные от App не сохраняются

    4.2. Сервер отправляет App новые, актуальные данные о вещи. App показывает пользователю сообщение, что данные вещи устарели. App обновляет данные в меню редактирования вещи

  **Бизнес правила:** 

    1. Право управления доступом к вещи имеет только владелец

    2. Владелец имеет право просматривать, изменять, удалять вещи общего доступа

    3. Друг имеет право изменять и просматривать вещи общего доступа

5. Func req 

  **ID: LS-1**

  **Название:** Просмотр списка доступных вещей

  **Акторы:** 

    1. App

    2. Пользователь

    3. Сервер

  **Предусловия:** 

    1. У пользователя есть интернет подключение

  **Основной поток:** 

    1. Пользователь открывает App и видит окно списка доступных вещей

    2. App отправляет запрос синхронизации на Сервер

    3. Сервер синхронизирует вещи общего доступа пользователя

    4. Сервер отправляет App ответ — данные доступных вещей из хранилища пользователя

    5. App отображает полученные данные вещей в списке доступных вещей

  **Постусловия:** 

    Успех:

      1. Данные о доступных пользователю вещах выгружены из хранилища сервера на App 

      2. Пользователь видит список доступных вещей и их информацию в карточках

    Неудача:

      1. Список доступных вещей не обновлён, взаимодействие с карточками вещей заблокировано. Пользователь видит сообщение "Ошибка синхронизации"

  **Альтернативные потоки:** 

    У пользователя нет сохранённых вещей

    Точка входа: Шаг 2

    2.1. Сервер отвечает сообщением об отсутствии сохранений вещей в хранилище пользователя

    2.2. App показывает пользователю сообщение, что у него нет сохранённых вещей. Список вещей пуст

  **Исключительные потоки:** 

    Сервер недоступен

    Точка входа: шаг 6

    6.1. App не получает ответа от сервера

    6.2. App выводит пользователю сообщение: "Сервер временно недоступен, попробуйте позже"

    6.3. App переходит на окно просмотра списка предметов

## Нефункциональные требования

### 1. Безопасность (Security)

- **ID:** SEC-01 ****

- **Название:** Зашифрованное хранение данных

- **Описание:** 3D-модели помещений и фотографии предметов должны храниться в зашифрованном виде в хранилище пользователя на сервере и локально на устройстве. На сервер передаются, в том числе обезличенные метаданные для аналитики.

- **Приоритет:** Высокий

- **Обоснование:** Если сервер взломают, злоумышленники не получат фото вашего дома, вещей и документов.



- **ID:** SEC-02:

- **Название:** Защита общего доступа

- **Описание:** Доступ к общему каталогу предметов должен требовать подтверждения (биометрия устройства или пин-код) при попытке просмотра или редактирования.

- **Приоритет:** Средний

- **Обоснование:** Если телефон потеряет член семьи, посторонний не увидит, где лежат ключи или документы.

### 2. Производительность (Performance)

- **ID:** PER-01

- **Название:** Скорость отклика при поиске

- **Описание:** Время от момента выбора предмета в каталоге до отображения его позиции на 3D-модели не должно превышать 2 секунд.

- **Приоритет:** Высокий

- **Обоснование:** Напрямую влияет на ключевую метрику продукта — время поиска вещи.



- **ID:** PER-02**:** 

- **Название:** Стабильный FPS

- **Описание:** При 3D-сканировании и визуализации карты частота кадров должна быть не ниже 30 FPS.

- **Приоритет:** Средний

- **Обоснование:** Низкая частота кадров делает приложение непригодным, пользователь не сможет точно рассмотреть положение предмета.



- **ID:** PER-03

- **Название:** Пропускная способность API

- **Описание:** API бэкенда должен обрабатывать не менее 500 RPS к ML-сервисам с временем ответа не более 5 секунд.

- **Приоритет:** Средний

- **Обоснование:** Обеспечение производительности в часы пик, когда много пользователей одновременно ищут вещи.

### 3. Надёжность (Reliability)

- **ID:** REL-01

- **Название:** Устойчивость к аварийному закрытию

- **Описание:** Если приложение вылетит во время 3D-сканирования, уже построенная часть модели должна быть сохранена и доступна при следующем запуске.

- **Приоритет:** Высокий

- **Обоснование:** Потеря 20 минут сканирования из-за случайного сбоя — худший пользовательский опыт, ведущий к удалению приложения.



- **ID:** REL-02

- **Название:** Бесперебойность распознавания

- **Описание:** Uptime сервиса определения места и предмета по фото должен не менее 99.5%.

- **Приоритет:** Высокий

- **Обоснование:** Это ядро продукта. Без ML нельзя добавить вещь или найти её.

### 4. Ёмкость (Capacity)

- **ID:** CAP-01

- **Название**: Ограничение занимаемого места на устройстве

- **Описание**: Приложение не должно занимать более 500 МБ постоянной памяти устройства (внутреннее хранилище) при сохранении типовой конфигурации: помещение на 100 м2 и 20 добавленных предметов с фотографиями.

- **Приоритет**: Высокий

- **Обоснование**: Пользователи не готовы жертвовать гигабайтами памяти ради приложения. Если приложение "раздуется", они его удалят. Также это снижает риск негативных отзывов



- **ID:** CAP-02

- **Название:** Потребление оперативной памяти

- **Описание:** Приложение в активном режиме (3D-сканирование/навигация) должно потреблять не более 500 МБ ОЗУ.

- **Приоритет:** Средний

- **Обоснование:** Обеспечение стабильной работы на среднебюджетных устройствах без выгрузки приложения системой и троттлинга (учёт риска "Ограничения аппаратного обеспечения").

### 5. Наблюдаемость (Observability)

- **ID:** OBS-01

- **Название:** Аналитика использования

- **Описание:** Приложение должно собирать обезличенные данные для расчета KPI: время поиска предмета, частота использования, точность позиционирования.

- **Приоритет:** Высокий

- **Обоснование:** Без этих данных невозможно измерить достижение бизнес-целей (MAU, Retention, время поиска).



- **ID:** OBS-02

- **Название:** Мониторинг ошибок

- **Описание:** В приложение должна быть встроена система сбора логов о падениях с фильтрацией по версии и модели устройства.

- **Приоритет:** Высокий

- **Обоснование:** Позволит команде быстро реагировать на проблемы с конкретными моделями телефонов (риск аппаратных ограничений).

### 6. Удобство использования (Usability)

- **ID:** USR-01

- **Название:** Обучение при первом запуске

- **Описание:** Процесс первого запуска должен содержать интерактивные подсказки по сканированию комнаты и добавлению первого предмета, занимающие не более 2 минут.

- **Приоритет:** средний

- **Обоснование:** Технология 3D-сканирования сложна для неопытного пользователя. Без онбординга он не начнет пользоваться продуктом.



- **ID:** USR-02

- **Название:** **Скорость добавления предмета**

- **Описание:** Процесс добавления нового предмета не должен занимать более 20 секунд.

- **Приоритет:** Средний

- **Обоснование:** Если процесс долгий, пользователь добавит 2-3 вещи и забросит приложение.

### 7. Поддерживаемость (Maintainability)

- **ID:** MAN-01

- **Название:** Независимость компонентов

- **Описание:** ML-модели должны быть выделены в отдельные сервисы с четким API, чтобы их можно было обновлять и дообучать независимо от основного приложения.

- **Приоритет:** Высокий

- **Обоснование:** Модели распознавания будут улучшаться. Возможность выкатить новую модель без пересборки всего приложения экономит ресурсы.



- **ID: MAN-02**

- **Название:** Документация API

- **Описание:** Все методы взаимодействия клиента и сервера должны быть задокументированы (Swagger/OpenAPI).

- **Приоритет:** Средний

- **Обоснование:** В команде бэкенд и Android-разработчик. Документация — их единственный способ синхронизации.

## Непокрытые категории НФТ

### 1. Совместимость (Compatibility)

- На этапе MVP используем главный сегмент ЦА - Android с поддержкой ARCore (согласно версии v1.0). Сознательно сужаем совместимость, чтобы сфокусировать ресурсы команды.

- **Риски:** Мы отсекаем рынок iOS и владельцев старых Android-смартфонов без ARCore.

- **План:** Добавить поддержку iOS (ARKit) в версии v2.0.

### 2. Масштабируемость (Scalability)

- Проектирование системы с горизонтальным масштабированием ML-сервисов (CAP-01 подразумевает это). Для MVP достаточно доказать, что схема работает. Детальные требования к автоматическому расширению кластеров пока избыточны.

- **Риски:** При резком успехе и взрывном росте пользователей может не хватить мощностей, пока DevOps вручную не добавит серверы.

- **План:** Внедрить auto-scaling на основе нагрузки (CPU/RPS) после стабилизации MVP и прогнозирования роста.

### 3. Нормативно-правовые требования (Legal & Regulatory)

- На этапе MVP основное регулирование — это 152-ФЗ (о персональных данных) и политика конфиденциальности. Юридическое сопровождение заложено в бюджет (п.9), но формулировать жесткие НФТ на этом этапе преждевременно — они появятся перед выходом на международный рынок или при изменении закона.

- **Риски:** Штрафы за утечку даже минимальных данных (email при регистрации), если они будут храниться на серверах без соответствующей защиты.

- **План:** Разработать и внедрить требования к защите персональных данных (соответствие 152-ФЗ) до публичного релиза (этап подготовки к v1.0).
